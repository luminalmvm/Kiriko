# Kiriko stack research — findings (2026-07)

Scope: Windows-first, GPU-accelerated After Effects alternative; solo/AI-assisted dev; developed partly on macOS, ships on Windows. Evidence gathered via web research from shipped apps (Rerun, Gyroflow, Cap, Graphite, Zed) and post-mortems (Olive, Natron).

---

## 1. Language: Rust vs C++ vs C#(+native core)

**Verdict: Rust.** The strongest evidence is that every recent *successful* indie media/creative app in this space is Rust, while the C++ indie precedents died of maintenance burden (see §6).

- **Concurrency**: Gyroflow's team explicitly credits Rust — "fearless concurrency, memory safety… allowed Gyroflow to grow very fast and avoid any memory issues and most crashes… while using a lot of threading and supporting all possible graphics APIs" ([Gyroflow used-technologies](https://docs.gyroflow.xyz/app/technical-details/used-technologies)). A compositor render graph is exactly the heavily-multithreaded workload where C++ data races are the classic failure mode; for a solo dev with no QA team this matters more than for a staffed team.
- **C interop**: Rust's C FFI is zero-cost and mature. Everything Kiriko needs from C (libav*, OFX, CUDA driver API, platform APIs) is reachable via bindgen. OFX hosting is *defined* as a C ABI (`OfxHost` struct + `fetchSuite`), so C++'s "native" advantage there is only the optional C++ support-layer convenience library; `openfx-sys` already provides raw bindings and `ofx-rs` demonstrates the plugin side ([openfx-sys](https://docs.rs/openfx-sys/latest/openfx_sys/), [ofx-rs](https://github.com/itadinanta/ofx-rs)).
- **C++ counterpoints**: bigger graphics/media sample ecosystem (NVIDIA Video Codec SDK samples, Qt, OpenColorIO/OpenImageIO are C++-first), and OCIO/OIIO bindings for Rust are thin. If deep OCIO integration becomes core, expect to write some C++ shim glue regardless of host language.
- **C# + native core**: real precedent — Olive's original author rebooted in 2025 "rewriting the node compositing system in C# … adopting the Godot engine" for dev-velocity reasons ([Olive status](https://www.olivevideoeditor.org/), Grokipedia summary). But C#'s GPU story (CUDA, D3D12 interop, custom compute) is all P/Invoke friction, GC pauses threaten frame pacing, and the macOS dev story is weaker. It optimises for the wrong constraint here (hiring/velocity for OO app code) when the hard part of Kiriko is the engine.
- **Build times**: real Rust cost. Mitigations: workspace of small crates (Graphite's structure), `mold`/`lld` linkers, sccache, cranelift backend for dev builds, keep `wgpu`/`ffmpeg-sys` in rarely-touched leaf crates. Incremental rebuilds of app-level crates stay in seconds; clean builds are minutes. Acceptable solo; would be a pain point for CI-heavy teams (irrelevant here).
- **Memory safety of deps**: ffmpeg stays C either way; Rust doesn't fix libav CVEs, but it does confine unsafety to the FFI boundary crate.

## 2. GPU abstraction

**Verdict: wgpu, with an explicit `as_hal` escape hatch for platform interop. This is the single biggest enabler of the "develop on macOS, ship on Windows" requirement** — one codebase runs DX12 on Windows and Metal on macOS with zero porting.

- **Overhead**: maintainers estimate wgpu at ~5–10% over raw Vulkan in real apps (validation + state/lifetime tracking) ([wgpu discussion #2080](https://github.com/gfx-rs/wgpu/discussions/2080)). For a compositor bottlenecked on bandwidth/decode, not draw-call count, this is noise.
- **Feature gaps (honest list)**: no bindless yet (proposal stage, "likely 2025–2026"), mesh shaders/raytracing native-only or absent, multi-queue absent ([HN](https://news.ycombinator.com/item?id=40609889), [WebGPU roadmap](https://kaelan.fyi/research/webgpu-future-roadmap/)). Native-extension features that *do* exist: subgroups, `shader-f16`, push constants, 64-bit atomics. A 2D compositor needs none of the missing ones badly; bindless would help a huge-asset node graph but texture arrays + bind group caching suffice at v1.
- **Production proof**: Rerun (viewer built on custom `re_renderer` over wgpu, [ARCHITECTURE.md](https://github.com/rerun-io/rerun/blob/main/ARCHITECTURE.md)); Bevy; Graphite (wgpu 29 + WGSL); **Cap.so ships a commercial screen recorder doing GPU colour-space conversion in WGSL over (a fork of) wgpu** ([Cap repo](https://github.com/CapSoftware/cap)); Gyroflow uses wgpu as one of its GPU compute backends in a shipping pro-video tool.
- **CUDA coexistence**: yes, in one process. Pattern documented in [wgpu discussion #7988](https://github.com/gfx-rs/wgpu/discussions/7988): run wgpu on the **Vulkan backend on Windows** when CUDA interop is needed, use `as_hal` to reach the raw Vulkan device, share memory via `VK_KHR_external_memory` → `cudaImportExternalMemory`, and sync with the new `vulkan::Queue::add_wait_semaphore` API (external-semaphore support landed in wgpu specifically for CUDA/OpenCL producers — see wgpu CHANGELOG). Working example: [oidn-wgpu-interop](https://github.com/Vecvec/oidn-wgpu-interop) (shares buffers with Intel OIDN the same way). Host-side CUDA from Rust: `cudarc` (safe driver-API bindings, actively maintained, used by candle) — kernels written in CUDA C++ compiled to PTX, or the rebooted Rust-CUDA project. NVENC/NVDEC are most practically reached **through ffmpeg's hwaccel** (`h264_nvenc`, `hevc_nvenc`, CUDA hwdevice) rather than raw Video Codec SDK bindings.
- **Hardware decode → texture (the sharp edge)**: wgpu has **no first-class texture-import API yet** ([issue #2320](https://github.com/gfx-rs/wgpu/issues/2320) still open); Linux got `texture_from_dmabuf_fd`, Windows D3D12/Vulkan import goes through `from_hal` with unsafe glue, and third-party [wgpu-graft](https://github.com/mark-ik/wgpu-graft) exists for GL FBO/Vulkan image/Metal IOSurface embedding. Realistic v1 plan: ffmpeg `d3d11va`/`d3d12va` (Windows) and `videotoolbox` (macOS) decode to GPU NV12, then **one GPU→GPU copy** into a wgpu texture + WGSL NV12→RGB pass (exactly what Cap does with its WGSL conversion shaders). Zero-copy is an optimisation project, not a v1 requirement — Gyroflow proves full zero-copy decode→process→display is *possible* in Rust but they lean on Qt RHI for it.
- **Raw Vulkan + MoltenVK**: maximal control (incl. Vulkan Video decode), but MoltenVK on macOS is a porting-layer dev experience (no Vulkan Video there anyway), and you hand-write all sync. Solo cost is very high; rejected.
- **DX12-only**: best Windows perf ceiling and simplest decode interop, but kills the macOS dev loop entirely (WARP/VM unusable for a creative app). Rejected on the stated constraint.
- **Colour-accurate viewport**: wgpu supports `Rgba16Float` / `Rgb10a2Unorm` surfaces; HDR/wide-gamut swapchain support on Windows exists but is less turnkey than raw DXGI — flag as an area needing a spike (scRGB swapchain via Vulkan backend or hal access).

## 3. UI toolkit

**Verdict: egui (+ eframe/winit) with `egui_dock` or Rerun's `egui_tiles`, custom-themed; viewport/timeline drawn via custom wgpu passes (re_renderer pattern). Runner-up worth a 1-week spike: GPUI.**

- **egui — evidence for**: Rerun is proof at scale — an entire professional data-visualisation product (docking workspace, timelines over millions of points, 3D viewports) written in immediate-mode egui, with `egui_wgpu` letting custom wgpu passes render inside UI panels ([Rerun ARCHITECTURE.md](https://github.com/rerun-io/rerun/blob/main/ARCHITECTURE.md)). Multi-window/detached panels: real native multi-viewport shipped in egui 0.24 (deferred + immediate viewports) ([release](https://github.com/emilk/egui/releases/tag/0.24.0)). Docking: `egui_dock` (drag-out into native windows, resizing, customisable look) and Rerun's own `egui_tiles`. Accessibility: AccessKit integrated, native APIs on Windows/macOS ([HN](https://news.ycombinator.com/item?id=33859697)). Theming: egui styling is fully programmatic (Rerun's design system looks nothing like stock egui), which suits a strict bespoke design language better than fighting a widget theme engine.
- **egui — honest cons**: text rendering is its historic weak spot — grayscale AA only, past kerning problems (kerning notably improved in 0.33 as it migrates toward parley, [release notes](https://github.com/emilk/egui/releases/tag/0.33.0)); no subpixel AA, so small text on low-DPI Windows monitors is worse than Qt/browser. Immediate mode costs CPU every frame; Rerun's own docs admit it "will only take us so far" and they add caching/background threads. A 10k-clip timeline must be culled/virtualised by hand (draw only visible items — straightforward in immediate mode, and Rerun's time panel does exactly this). Complex text input (IME) historically rough.
- **GPUI (Zed's framework)**: hybrid immediate/retained, GPU-accelerated, now genuinely cross-platform — Zed shipped Windows support (Jan 2026 stable rollout after a hard port; they swapped their Blade renderer for **DirectX 11** on Windows due to driver/ARM64 issues — [The Register](https://www.theregister.com/2025/08/22/everything_is_different_on_windows/), [Windows Forum analysis](https://windowsforum.com/threads/zed-windows-port-the-hidden-cost-of-native-gpu-ui-in-cross-platform-dev.378446/)). Ecosystem growing ([awesome-gpui](https://github.com/zed-industries/awesome-gpui), [gpui-component](https://github.com/longbridge/gpui-component) with a Dock/tiles layout). Better text rendering than egui, retained-mode perf. Cons: documentation thin, API churns with Zed's needs, no crates.io stability story until recently, embedding your *own* wgpu viewport inside GPUI is less proven than `egui_wgpu`. Worth a spike; risky as the initial bet.
- **Iced**: Elm architecture, no mature docking story, slower iteration for deeply custom widgets; general 2025 consensus positions it for structured business apps, not canvas-heavy pro tools ([comparison](https://an4t.com/rust-gui-libraries-compared/)).
- **Slint**: royalty/commercial licence for proprietary desktop apps, DSL-driven, widget-set oriented — wrong shape for a bespoke timeline/graph-editor app.
- **Qt (cxx-qt or C++)**: the *proven* pro-media answer — Gyroflow ships Rust core + QML UI with zero-copy Qt RHI preview. But: LGPL/commercial licensing tax, CMake/qmake toolchain bolted onto cargo, cxx-qt still maturing ("not as mature as PyQt… on the way" — [KDAB](https://www.kdab.com/cxx-qt/)), QML styling fights a bespoke design system, and two-language cognitive load. Gyroflow proves it works; it's just more moving parts than egui for a solo dev.
- **Dear ImGui (C++)**: docking branch is excellent and battle-tested in game tooling, but drags the project to C++ (see §1) and has the same text-rendering class of issues as egui.
- **Fully custom on wgpu + Vello + cosmic-text (Graphite path)**: Vello and Xilem are still self-described **alpha** ([Linebender](https://linebender.org/blog/tmil-24/)); note Graphite's actual shipped UI is **Svelte in a web view / CEF shell**, not a Rust-native toolkit — i.e. even Graphite deferred the custom-Rust-UI problem. Building a toolkit is a multi-year detour; rejected.
- **DPI/multi-monitor on Windows**: winit handles per-monitor DPI v2; egui inherits it. Zed's port story is the cautionary tale that Windows rendering/driver diversity always costs extra weeks — budget for it whatever the toolkit.

## 4. Media layer

- **Bindings**: **rsmpeg** (larksuite) over ffmpeg-next — actively maintained, tracks FFmpeg 6/7/8 (`rsmpeg 0.18+ffmpeg.8.0`), "thin and safe layer" philosophy ([rsmpeg](https://github.com/larksuite/rsmpeg)); ffmpeg-next is explicitly **maintenance-only** ([crates.io](https://crates.io/crates/ffmpeg-next)). Cap maintains its own rust-ffmpeg fork — expect to vendor/patch bindings either way; design an internal `media` crate boundary so the binding choice is swappable. Link libav dynamically (LGPL compliance + easy codec upgrades); GPL build variant optional for x264 export.
- **Hardware pipelines**: decode via avcodec hwaccel — `d3d11va`/`d3d12va` → `ID3D11Texture2D` NV12 array slices on Windows (standard mpv/Firefox pattern), `videotoolbox` → `CVPixelBuffer`/IOSurface on macOS; encode via `*_nvenc` (NVIDIA), `*_amf` (AMD), `*_qsv` (Intel), `videotoolbox` (macOS). ROCm has no NVENC-equivalent Rust story; on AMD go through AMF via ffmpeg, and do compute in WGSL (portable) rather than HIP.
- **Audio**: **cpal** is the standard but has real A/V-sync gaps: no device-latency query API ([issue #1154](https://github.com/RustAudio/cpal/issues/1154), [#279](https://github.com/RustAudio/cpal/issues/279)), WASAPI timing quirks. For an editor this is manageable: master clock = audio samples consumed (count samples written in the callback), video slaves to it — the mpv approach; editors need *stable* sync, not 5 ms live latency. Fallback: talk to WASAPI directly via the `wasapi`/`windows` crates on Windows for `GetStreamLatency`. Cap ships on cpal.
- **Demux/containers**: libavformat covers everything; pure-Rust `symphonia` is good for audio-only decode paths.

## 5. Windowing & shell

- **winit**: de-facto standard, Windows backend mature, multi-window supported (egui's multi-viewport rides on it); 1.0 targeted then maintenance mode ([winit](https://github.com/rust-windowing/winit)). Known cost: event-loop-owns-app architecture (ApplicationHandler) is awkward but fine under eframe.
- **Menus**: winit deliberately excludes native menus → **muda** (used by Tauri) for Windows/macOS native menu bars.
- **File dialogs**: **rfd** — native dialogs, async-friendly with winit; enable the ComCtl32 v6 feature for modern Windows message dialogs ([rfd docs](https://docs.rs/rfd/latest/rfd/)).
- These are exactly the shell pieces Cap/Rerun already ship on.

## 6. Precedents — what helped and hurt

| Project | Stack | Outcome / lesson |
|---|---|---|
| **Rerun** | Rust, egui + custom re_renderer on wgpu, eframe | Shipped, VC-funded, proves egui+wgpu can carry a pro docking/timeline/viewport product. Lesson: build a thin renderer crate over wgpu; cull aggressively; immediate mode needs deliberate caching at scale. |
| **Gyroflow** | Rust core, QML UI, wgpu/OpenCL compute, ffmpeg (mdk-sdk) | Shipped pro video tool with zero-copy GPU pipeline. Lesson: Rust core + heavy threading is stable in production; Qt RHI made zero-copy easy but adds a C++/QML layer. |
| **Cap (cap.so)** | Rust, Tauri v2 + Solid UI, forked wgpu + rust-ffmpeg, cpal, WGSL colour conversion | Shipped commercial recorder/editor. Lesson: the Rust media pipeline (wgpu+ffmpeg+cpal) is production-viable today, but expect to maintain forks of bindings. |
| **Graphite** | Rust core → wasm, **Svelte UI in CEF shell**, wgpu/Vello raster | Shipped web-first. Lesson: even the flagship Rust creative app punted on Rust-native UI; node-graph-as-Rust-workspace with proc-macro node registration is a strong architecture to copy. |
| **Zed** | Rust, in-house GPUI (Metal → Blade → **D3D11 on Windows**) | Shipped, but the Windows port took years and a renderer swap. Lesson: Windows GPU/driver diversity is where cross-platform GPU UIs bleed; DX-family backends are safer than Vulkan on consumer Windows. |
| **Olive** (C++/Qt NLE) | C++, Qt, OpenGL | Development halted Sep 2023; solo-dev burnout, crashy nightlies, "expansive goals"; author rebooted in **C#/Godot** for velocity ([status](https://www.olivevideoeditor.org/)). Lesson: solo C++ app scope-death is the base rate; keep v1 scope brutal. |
| **Natron** (C++ OFX compositor — the literal AE-alternative precedent) | C++98, Qt, OFX host | Died when Inria funding and 2 core devs left (2018) ([Libre Arts post-mortem](https://librearts.org/2018/11/the-demise-of-natron/)). Lessons: OFX hosting was its superpower (whole plugin ecosystem for free) — Kiriko hosting OFX is high-leverage; bus-factor-1 C++ codebases don't survive maintainer loss. |

## 7. Recommended stack + strongest counter-argument

**Recommended:**
- **Language**: Rust (edition 2024), cargo workspace of small crates (Graphite-style: `engine`, `render`, `media`, `ui`, `ofx-host`); dev builds with lld/mold + sccache.
- **GPU**: wgpu — DX12 backend default on Windows, Metal on macOS; **switch to the Vulkan backend on Windows when CUDA features are enabled** so `as_hal` + `VK_KHR_external_memory/semaphore` + `cudarc` give CUDA interop; all first-party effects in WGSL compute so every GPU vendor (incl. AMD, no ROCm needed) gets acceleration; CUDA kernels as optional fast paths only.
- **UI**: egui + eframe, `egui_tiles`/`egui_dock` docking, multi-viewport for detached panels, AccessKit on; timeline/curve/graph editors as custom egui widgets with hand-culled drawing; colour-accurate monitor as a custom wgpu pass composited via `egui_wgpu` (Rerun's re_renderer pattern — build `kiriko_renderer` as its own crate from day one).
- **Media**: system/vendored libav via **rsmpeg**; hwaccel decode (d3d11va/d3d12va, videotoolbox) with one GPU copy into wgpu + WGSL NV12/P010→linear conversion; encode via ffmpeg NVENC/AMF/QSV/VideoToolbox; **cpal** audio with audio-samples-consumed master clock.
- **Shell**: winit + muda (menus) + rfd (dialogs). OFX hosting via `openfx-sys`-style bindgen over the C ABI, on a dedicated render path (OFX effects are CPU/GL-oriented — plan a staging buffer path).

**Strongest counter-argument**: *the UI layer is the bet most likely to hurt.* egui's text rendering (no subpixel AA, only-recently-fixed kerning) and immediate-mode CPU burn are real, permanent taxes on a text-dense pro app judged against After Effects' polish, and no egui app has shipped a timeline UI at AE's interaction depth (Rerun's is simpler). The proven alternative for exactly this product shape is Gyroflow's Rust-core + Qt/QML shell, which buys pro-grade text, styling, docking, and zero-copy presentation at the price of C++/QML glue, licensing, and a second toolchain. Mitigation: the workspace split (engine/renderer/UI in separate crates) keeps the UI swappable — egui for v1 velocity, with GPUI or QML as the escape hatch if polish stalls; secondary risks are wgpu's missing texture-import API (one extra GPU copy per decoded frame until it lands) and wgpu's bindless gap if node graphs grow past a few hundred live textures.
